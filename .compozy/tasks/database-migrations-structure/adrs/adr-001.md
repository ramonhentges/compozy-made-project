# ADR-001: Migrations Structure for Bounded Contexts

## Status
Accepted

## Date
2026-04-18

## Context
We need to implement a database migration structure that supports independent deployment and versioning of database changes per bounded context/module. The current approach likely has all migrations in a single folder, causing conflicts and coupling between teams working on different bounded contexts.

## Decision
We will implement a separated migrations structure where each bounded context has its own migration folder under `src/migrations/<bounded-context-name>/`. This approach:
- Enables independent deployment and versioning of database changes per module
- Reduces migration conflicts between teams
- Provides clear ownership of database changes per bounded context
- Aligns with the existing codebase organization by feature/domain directories

## Alternatives Considered
1. **Single migrations folder**: All migrations in `src/migrations/` - leads to conflicts and coupling
2. **Migrations inside each bounded context**: `src/modules/<context>/migrations/` - couples migration lifecycle with module deployment
3. **Hybrid approach**: Shared configuration with separate folders - adds complexity without clear benefits

## Consequences
### Positive
- Teams can manage database changes independently per bounded context
- Reduced risk of migration conflicts during concurrent development
- Clear separation of concerns aligned with domain-driven design
- Easier to track database evolution per business capability

### Negative
- Slight increase in overall migration folder structure complexity
- Need to ensure consistent migration tooling configuration across contexts
- Potential duplication of migration utility scripts (mitigated by shared configuration)
