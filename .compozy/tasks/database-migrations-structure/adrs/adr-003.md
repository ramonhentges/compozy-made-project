# ADR-003: Shared Data Handling Approach for Separate Databases

## Status
Accepted

## Date
2026-04-18

## Context
With the decision to implement separate databases per bounded context, the approach to handling data that needs to be accessed by multiple bounded contexts must be reconsidered. True shared tables are not possible when each bounded context has its own physically separate database instance.

## Decision
We will handle cross-context data access through service APIs rather than direct database sharing. This means:
- Each bounded context owns its data exclusively in its own database
- When a bounded context needs data owned by another context, it accesses that data through the owning context's API
- For data that is truly shared and needs strong consistency, we will implement synchronous API calls
- For data that can tolerate eventual consistency, we will use event-driven architecture (domain events) to propagate changes
- We will not attempt to create shared tables or direct database access between contexts

## Alternatives Considered

### Alternative 1: Direct database access (shared tables via database links/replication)
- **Description**: Allow bounded contexts to directly access tables in other contexts' databases
- **Pros**: Familiar relational data access patterns, strong consistency
- **Cons**: Violates bounded context encapsulation, creates tight coupling, complex to manage permissions and connections
- **Why rejected**: Goes against the principle of bounded context autonomy; creates runtime coupling between contexts

### Alternative 2: Database replication for shared data
- **Description**: Replicate shared tables from owning context's database to consuming contexts' databases
- **Pros**: Each context has local copy for fast reads, reducing API latency
- **Cons**: Replication lag introduces inconsistency, complex setup and monitoring, potential for conflicts
- **Why rejected**: Overly complex for V1; eventual consistency issues; better handled through explicit APIs

### Alternative 3: Eventual consistency via messaging/events
- **Description**: Use events to propagate changes to shared data, accepting eventual consistency
- **Pros**: Maintains loose coupling, good performance, aligns with event-driven architecture
- **Cons**: Data may be temporarily inconsistent, requires conflict resolution strategies
- **Why rejected**: While this will be used for some scenarios, synchronous API access is needed for strongly consistent reads

### Alternative 4: Shared database for common data only
- **Description**: Keep separate databases per context but maintain a small shared database for truly common data
- **Pros**: Balances autonomy with practical sharing needs
- **Cons**: Still creates a coupling point, need to determine what qualifies as "truly common"
- **Why rejected**: Adds complexity; better to start with pure separation and add sharing mechanisms only when proven necessary

## Consequences

### Positive
- Maintains strong bounded context encapsulation and autonomy
- Eliminates risk of migration conflicts between contexts (no shared schema)
- Clear ownership and responsibility for data
- Enables independent scaling, backup, and maintenance of databases
- Aligns with microservices-like benefits even in modular monolith

### Negative
- Cross-context data access requires API calls, which may have higher latency than direct database queries
- Need to develop and maintain APIs for data access
- May require handling of partial failures in distributed data access patterns
- Data duplication may occur if same data is needed in multiple contexts for performance

## Implementation Notes
- Each bounded context will have its own database instance (e.g., identity_db, billing_db, product_catalog_db)
- Connection configurations will be specific to each context
- Migration folders remain per context and will only migrate that context's database
- For cross-context reads: Implement service APIs in each context to expose necessary data
- For cross-context writes: Use commands/events to request changes from the owning context
- Consider implementing a read-through cache pattern for frequently accessed cross-context data
- Document data ownership clearly: which context is the system of record for each piece of data
- Consider implementing a data access layer or facade to abstract whether data comes from local database or remote API
