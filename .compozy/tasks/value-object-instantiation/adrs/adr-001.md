# ADR-001: Value Object Instantiation Pattern

## Status

Accepted

## Date

2026-04-17

## Context

The current pattern for value objects uses private constructors with static factory methods (`create()`) that validate input. Mappers need to instantiate value objects from already-validated database values, but currently they must use the static factory methods which perform redundant validation. Additionally, ID-type value objects need auto-generation capability.

## Decision

The value object instantiation pattern will be:

1. **UserId**: Constructor is public. `create(value: string)` validates and throws if invalid. `create()` (no args) auto-generates a UUID v4.
2. **Email**: Constructor is public. `create(email: string)` validates and throws if invalid.
3. **Password**: Constructor is public. `create(hash: string)` validates and throws if empty.
4. **Mappers**: Use public constructors directly for already-validated DTO data (no re-validation needed).

## Alternatives Considered

### Alternative 1: Add separate new() method for generation

- **Description**: Keep create() as validation-only, add new() static method for auto-generation
- **Pros**: Clear separation of concerns
- **Cons**: Two methods to maintain, API inconsistency
- **Why rejected**: User wants unified create() method that handles both cases

### Alternative 2: Use constructor everywhere, remove static factory

- **Description**: Make constructors public, use them directly everywhere
- **Pros**: Simpler, no factory methods
- **Cons**: Loses validation at creation points (use cases), harder to enforce invariants
- **Why rejected**: Use cases need validation when creating new entities/value objects

### Alternative 3: Distinct naming per type

- **Description**: generateUserId(), createEmail(value), createPassword(value)
- **Pros**: Clear intent
- **Cons**: Inconsistent API across value objects
- **Why rejected**: Inconsistent API confusing for developers

## Consequences

### Positive

- Mappers can use constructors directly for better performance
- ID generation becomes consistent across all ID types
- Clear distinction: use cases use `create(value)`, mappers use constructors
- Backward compatible for use cases (create() still works)

### Negative

- All value objects need constructor visibility changed to public
- Requires updating all mappers across all modules

### Risks

- Need to ensure all instantiation goes through proper validation paths
- Test coverage needs to verify both constructor and create() paths work

## Implementation Notes

1. For each value object: make constructor public, add parameterless create() for IDs
2. Update all mappers to use constructors instead of create()
3. Keep create() for use case validation
4. Apply to all modules (not just identity)
