# ADR-004: Kafka Relay With In-Process Polling

## Status

Accepted

## Date

2026-04-21

## Context

The PRD requires asynchronous delivery, retry recovery, status visibility, and explicit at-least-once semantics. The current codebase has no message broker integration and starts as a single Fastify process. V1 should use a real broker integration while avoiding a broad eventing platform.

The selected broker is Kafka. The selected worker model is an in-process polling loop started by the existing Fastify application. The selected retry policy is exponential backoff with a configurable maximum attempt count and terminal `failed` status.

## Decision

V1 will publish outbox records to Kafka through an application-level publisher port and a Kafka infrastructure adapter. The relay will run as an in-process polling worker started during application bootstrap and stopped during graceful shutdown.

The outbox table will use `pending`, `processing`, `published`, and `failed` statuses. It will store retry metadata through `attempts`, `next_attempt_at`, `last_error`, and `published_at`. The relay will claim due `pending` records in batches, mark them `processing`, publish to Kafka keyed by `aggregateId`, mark successful records `published`, and reschedule transient failures with exponential backoff until the configured maximum attempt count is reached.

## Alternatives Considered

### Alternative 1: No-Op or Logging Publisher

- **Description**: Ship persistence and retry mechanics before adding a real broker.
- **Pros**: Smaller implementation and fewer runtime dependencies.
- **Cons**: Does not prove end-to-end delivery to a downstream system.
- **Why rejected**: The selected V1 should target a real broker integration.

### Alternative 2: PostgreSQL LISTEN/NOTIFY

- **Description**: Use PostgreSQL notification channels as the delivery mechanism.
- **Pros**: Avoids adding broker infrastructure.
- **Cons**: Does not match the selected Kafka destination and is weaker for durable downstream event streaming.
- **Why rejected**: Kafka is the chosen broker.

### Alternative 3: Separate Worker Process

- **Description**: Run a standalone Node worker or CLI process that polls and publishes independently from the API.
- **Pros**: Better operational isolation and horizontal scaling.
- **Cons**: Adds deployment complexity before V1 proves the pattern.
- **Why rejected**: The selected V1 worker model is an in-process loop with configurable interval and batch size.

## Consequences

### Positive

- V1 validates durable capture and real Kafka delivery together.
- The application port keeps Kafka details outside domain and use-case code.
- In-process polling minimizes deployment changes for the first release.
- Retry state remains inspectable in the database and logs.

### Negative

- The API process now owns background work and Kafka connectivity.
- Scaling multiple API instances requires safe record claiming.
- Kafka client configuration and lifecycle management become part of application bootstrap.

### Risks

- **Duplicate delivery**: Mitigate through stable event IDs, at-least-once documentation, and idempotent consumer requirements.
- **Concurrent relay claims**: Mitigate with transactional status changes and row-level locking where supported.
- **Poison records**: Mitigate with maximum attempts, terminal `failed` status, and structured failure logs.
- **Kafka outage affects process health**: Mitigate by retrying asynchronously and not blocking user-facing requests on Kafka publish.

## Implementation Notes

- Add Kafka configuration for brokers, client ID, topic, poll interval, batch size, and max attempts.
- Use a concrete Kafka client package for the infrastructure adapter.
- Publish Kafka records with `aggregateId` as the partition key so events for the same aggregate stay on the same partition.
- Start the relay after Fastify dependencies are composed.
- Stop the relay and disconnect the Kafka producer during graceful shutdown.
- Redact sensitive payload details from logs.
