# ADR-002: Entity Constructor Visibility

## Status

Accepted

## Date

2026-04-17

## Context

The User entity has a private constructor called via `User.create()`. Mappers need to reconstruct entities from database data without re-running business logic. The PRD requires making constructors public for value objects and entities.

## Decision

The User entity constructor will be made public directly, paralleling the value object pattern:
- Public constructor for reconstruction from database via mappers
- Static `create()` method retained for application layer validation
- Each parameter explicitly typed to enforce invariants

## Alternatives Considered

### Alternative 1: Add fromPersistence() method

- **Description**: Keep private constructor, add static `fromPersistence(dto)` method
- **Pros**: Clear separation of creation paths
- **Cons**: Duplicate pattern with value objects, inconsistent API
- **Why rejected**: User requested same pattern as value objects

### Alternative 2: Reconstruction method with different name

- **Description**: Add static `reconstruct()` method with different signature
- **Pros**: Intent is clear
- **Cons**: Additional method to maintain
- **Why rejected**: Public constructor is simpler and matches value object pattern

## Consequences

### Positive

- Consistent API with value objects
- Mappers use constructor directly
- Clear distinction: application layer uses `create()`, persistence uses constructor

### Negative

- Entity constructor must accept all parameters in correct order

### Risks

- Developers could bypass validation if they use constructor in application layer
- Mitigation: code review and linting rules

## Implementation Notes

1. Change constructor from `private constructor` to `public constructor`
2. Keep static `create()` method for use cases
3. Update UserMapper to use constructor instead of `create()`
