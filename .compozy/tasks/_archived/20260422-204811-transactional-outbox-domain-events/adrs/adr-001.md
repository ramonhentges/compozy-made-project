# ADR-001: Minimal Transactional Outbox Scope

## Status

Accepted

## Date

2026-04-21

## Context

The project already models domain events as pure domain objects collected by aggregate roots, but persistence currently writes aggregates directly through pg-promise repositories without a transaction boundary, event dispatcher, message relay, or outbox.

The feature idea targets a specific reliability problem: prevent lost domain events when aggregate persistence succeeds but event publishing fails. The user selected a minimal V1 scope, with product/application teams as the primary beneficiary and adoption across multiple workflows as the first success signal.

The main force is scope discipline. A broad eventing platform would slow the first release, but an underspecified "table plus worker" would create hidden reliability, security, and operability risks.

## Decision

V1 will define a minimal transactional outbox for domain events that atomically persists aggregate changes and their domain events in one database transaction, then publishes those outbox records asynchronously with explicit at-least-once delivery semantics.

The outbox is an application/infrastructure reliability mechanism, not a domain concern. Domain events remain pure facts raised by aggregates. Application use cases own event extraction and transactional persistence through explicit ports. Infrastructure owns the outbox table, pg-promise transaction implementation, relay, retry state, broker integration, and operational metadata.

V1 must include the semantic floor needed for safe adoption: stable event identity, event type/version envelope, aggregate identity, retry/status metadata, basic failure visibility, idempotency expectations for consumers, payload minimization, and a retention posture. It will not include a general event platform, advanced replay UI, broad broker abstraction, global ordering guarantees, or workflow orchestration.

## Alternatives Considered

### Alternative 1: Direct Publish After Commit

- **Description**: Persist the aggregate, then immediately publish domain events from the application handler.
- **Pros**: Lowest implementation effort and no additional persistence structure.
- **Cons**: Does not solve the selected problem; a process crash or broker failure after commit still loses the event.
- **Why rejected**: It preserves the dual-write failure mode the feature exists to remove.

### Alternative 2: Full Eventing Platform

- **Description**: Build a broad event bus abstraction with replay tooling, admin UI, multiple broker support, workflow orchestration, and advanced observability.
- **Pros**: More complete long-term platform and richer operations.
- **Cons**: Slower delivery, larger maintenance burden, and likely overbuild before adoption is proven.
- **Why rejected**: The first release should validate reliable event adoption across workflows before expanding into platform capabilities.

### Alternative 3: Minimal Table and Polling Worker Only

- **Description**: Add an outbox table and polling worker without formal delivery semantics, event envelope, status model, or consumer contract.
- **Pros**: Small and fast to implement.
- **Cons**: Creates duplicated, unordered, invisible, or unsafe workflow behavior under retry and partial failure.
- **Why rejected**: Minimal V1 should be small in machinery, not vague in guarantees.

## Consequences

### Positive

- Reduces lost domain events without distributed transactions.
- Preserves Clean Architecture by keeping database and broker concerns outside the domain.
- Creates a reusable reliability path that can compound as more workflows adopt domain events.
- Makes delivery semantics explicit early, reducing fragmentation across future workflows.

### Negative

- Requires introducing a transaction boundary where none exists today.
- Adds operational state that must be monitored and maintained.
- Requires consumers to handle at-least-once delivery and duplicates.
- Adds durable copies of event payloads that need retention and privacy controls.

### Risks

- **Duplicate side effects**: Mitigate through stable event IDs and documented idempotent consumer requirements.
- **Poison events block delivery**: Mitigate with retry limits and terminal failed/quarantined status.
- **Sensitive data leakage**: Mitigate with payload minimization, event allowlist, retention policy, and redacted logs.
- **Pattern fragmentation**: Mitigate with explicit application ports and one infrastructure adapter path.

## Implementation Notes

- Start in the Identity context because the existing `User` aggregate already raises `UserRegistered` and `PasswordChanged`.
- Add an outbox migration under `src/migrations/identity_context/`.
- Add an application-level transaction or unit-of-work boundary so repository writes and outbox inserts commit atomically.
- Keep relay publishing outside the request transaction.
- Treat delivery as at-least-once; do not promise exactly-once delivery.
