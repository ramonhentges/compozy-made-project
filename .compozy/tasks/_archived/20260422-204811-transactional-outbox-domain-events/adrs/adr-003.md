# ADR-003: Repository-Captured Outbox Persistence

## Status

Accepted

## Date

2026-04-21

## Context

The Identity domain already raises domain events from aggregate roots and stores them in memory until `pullDomainEvents()` is called. The current pg-promise repository writes aggregates directly and does not persist pending domain events. The transactional outbox must prevent events from being lost after a committed aggregate change while keeping domain objects free of infrastructure concerns.

The selected implementation must fit the existing Clean Architecture structure, where repositories are driven ports and pg-promise implementations live in infrastructure. It must cover aggregate insert, update, and delete operations.

## Decision

Identity repository persistence methods will persist pending aggregate domain events together with the aggregate root in the same database transaction. After the aggregate write and outbox inserts succeed, the repository implementation will pull the saved events from the aggregate root.

The repository port remains the application-facing persistence boundary. The infrastructure repository will open a pg-promise transaction for `save`, `update`, and `delete`, write the aggregate change, insert one outbox row per pending domain event, and clear the aggregate's in-memory events only after the transaction work has been accepted.

## Alternatives Considered

### Alternative 1: Application Unit of Work Port

- **Description**: Add a separate application-level `UnitOfWork` port that coordinates repository writes and outbox inserts.
- **Pros**: Makes transaction boundaries explicit at the use-case layer and can support multiple repositories later.
- **Cons**: Adds an abstraction before the codebase has cross-repository transactions.
- **Why rejected**: V1 only needs Identity aggregate persistence, so repository-captured events are smaller and align with the user's selected boundary.

### Alternative 2: Outbox-Specific Repository Methods

- **Description**: Add methods such as `saveWithEvents(user)` beside normal persistence methods.
- **Pros**: Makes outbox behavior explicit in method names.
- **Cons**: Creates parallel persistence APIs and makes it easy to call the non-outbox path accidentally.
- **Why rejected**: All Identity aggregate persistence should capture events consistently by default.

### Alternative 3: External Handler Wrapper

- **Description**: Wrap existing application handlers in infrastructure and capture events outside repository persistence.
- **Pros**: Avoids changing repository implementations.
- **Cons**: Hides transaction behavior outside the application contract and risks coupling HTTP/bootstrap code to domain internals.
- **Why rejected**: It weakens the persistence boundary and makes testing the transaction contract harder.

## Consequences

### Positive

- Aggregate changes and outbox rows commit atomically.
- Application handlers can keep using existing repository methods.
- Domain events remain pure domain objects.
- The implementation stays focused on the Identity bounded context.

### Negative

- Repository methods now own transaction orchestration and outbox persistence.
- Tests must assert that domain events are only pulled after successful persistence.
- Future cross-aggregate transactions may still require a separate unit-of-work abstraction.

### Risks

- **Events cleared before rollback**: Mitigate by pulling events only after outbox insert work succeeds, and by testing failed transaction paths.
- **Repository responsibility growth**: Mitigate by extracting only infrastructure helpers for outbox row mapping and SQL, not a new domain abstraction.
- **Delete event ambiguity**: Mitigate by persisting pending events already raised by the aggregate before deleting the row; V1 does not invent new delete events.

## Implementation Notes

- Use the existing `AggregateRoot.pullDomainEvents()` contract.
- Add outbox persistence under `src/modules/identity/infrastructure/persistence/`.
- Add an Identity migration for the `events` outbox table under `src/migrations/identity_context/`.
- Keep outbox row mapping infrastructure-only.
