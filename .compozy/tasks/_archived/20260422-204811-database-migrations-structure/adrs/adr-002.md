# ADR-002: Migration Tooling Selection

## Status
Accepted

## Date
2026-04-18

## Context
The project currently has no database migrations structure in place. We need to select a migration tooling/framework that will work well with the bounded contexts approach and integrate with the existing Node.js/TypeScript codebase. The selection should consider ease of use, community support, and compatibility with the separated migrations per bounded context approach.

After initial evaluation of TypeORM, the alternative node-pg-migrate was brought to attention and evaluated as well.

## Decision
We will use node-pg-migrate as the migration tooling framework. This decision is based on:
- Specifically designed for PostgreSQL with excellent support
- Lightweight and focused purely on migrations (no ORM overhead)
- Excellent CLI support with rich feature set
- Works perfectly with multiple migration folders (supports --migrations-dir parameter)
- Native TypeScript support
- Simpler to understand and use than full ORM migration systems
- Active maintenance and good documentation
- Does not require adopting an ORM throughout the application
- Well-suited for the separate database per bounded context approach

## Alternatives Considered

### Alternative 1: TypeORM migrations
- **Description**: Using TypeORM's migration capabilities
- **Pros**: Familiar to Node.js/TypeScript developers, good CLI, potential future ORM integration
- **Cons**: Heavier weight (includes ORM even if not used), more complex configuration, overkill if only needing migrations
- **Why rejected**: node-pg-migrate is more focused and lightweight for pure migration needs; avoids unnecessary ORM complexity

### Alternative 2: Sequelize CLI
- **Description**: Using Sequelize's migration CLI tool
- **Pros**: Well-established, good documentation, works with multiple databases
- **Cons**: More verbose migrations, less ideal if not using Sequelize ORM, additional complexity
- **Why rejected**: More verbose than node-pg-migrate; Sequelize migrations require more boilerplate

### Alternative 3: Flyway
- **Description**: Using Flyway for database migrations
- **Pros**: Database-agnostic, strong versioning capabilities, works well with Spring/Java ecosystem
- **Cons**: Primarily Java-focused, less native Node.js/TypeScript integration, requires JVM
- **Why rejected**: Overkill for Node.js project; introduces JVM dependency unnecessarily

### Alternative 4: Prisma Migrate
- **Description**: Using Prisma's migration system
- **Pros**: Excellent developer experience, automatic migration generation, good TypeScript support
- **Cons**: Requires adopting Prisma ORM throughout the application, couples migration tool with ORM choice
- **Why rejected**: Would require migrating entire data access layer to Prisma, which is beyond the scope of this V1 implementation

### Alternative 5: Custom migration scripts
- **Description**: Creating custom Node.js scripts to handle migrations
- **Pros**: Full control, no external dependencies
- **Cons**: Significant development effort, lack of standard features (version tracking, rollbacks, etc.)
- **Why rejected**: Re-inventing the wheel when established solutions like node-pg-migrate exist

## Consequences

### Positive
- Lightweight and focused purely on migration concerns
- Excellent PostgreSQL-specific features and optimizations
- Simple, intuitive migration API
- Rich CLI with many options (schema support, locking, transactions, etc.)
- Works perfectly with multiple migration folders via --migrations-dir
- Native TypeScript support with .ts migration files
- Active community and maintenance
- No ORM overhead or unwanted dependencies
- Easy to understand and adopt

### Negative
- PostgreSQL-only (not database agnostic) - but we're using PostgreSQL
- Smaller community than TypeORM (but still healthy)
- Fewer IDE integrations compared to larger frameworks

### Risks
- If we need to support multiple databases in the future, we'd need to switch
  *Mitigation*: Currently committed to PostgreSQL per bounded context; node-pg-migrate excels at this
- Less known than TypeORM in some circles
  *Mitigation*: Documentation and training can overcome this; framework is straightforward to learn

## Implementation Notes
- Install node-pg-migrate as dev dependency: `npm install -D node-pg-migrate`
- Create base config that can be extended per context (or use environment variables)
- Each bounded context will have its own migration folder and database connection
- Migration commands will be added to package.json for each context (e.g., `migrate:identity_context`)
- Supports TypeScript migration files natively
- Can use pg-migrate programmatic API if needed for custom integrations
