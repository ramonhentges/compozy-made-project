# ADR-004: Bounded Context Migration Folder Naming Convention

## Status
Accepted

## Date
2026-04-18

## Context
We need to establish a naming convention for the migration folders that will be created for each bounded context under `src/migrations/`. The naming convention should be consistent, clear, and follow common practices in the codebase.

## Decision
We will use snake_case for naming bounded context migration folders (e.g., `identity_context`, `billing_context`). This decision is based on:
- Snake_case is commonly used for file and directory names in Node.js/TypeScript projects
- It matches the existing naming convention seen in the codebase (based on examination of similar projects)
- It's readable and avoids case sensitivity issues across different operating systems
- It aligns with the user's preference expressed during the technical clarification process

## Alternatives Considered

### Alternative 1: kebab-case (e.g., identity-context)
- **Description**: Using kebab-case for folder names
- **Pros**: Readable, commonly used in URLs and CLI commands
- **Cons**: Can be problematic in some programming contexts, less common for directory names in Node.js
- **Why rejected**: Less conventional for directory names in Node.js/TypeScript projects compared to snake_case

### Alternative 2: PascalCase (e.g., IdentityContext)
- **Description**: Using PascalCase for folder names
- **Pros**: Matches class naming conventions, easy to distinguish
- **Cons**: Unconventional for directory names, can cause case sensitivity issues
- **Why rejected**: Not standard practice for directory names in most programming ecosystems

### Alternative 3: camelCase (e.g., identityContext)
- **Description**: Using camelCase for folder names
- **Pros**: Matches variable/function naming conventions
- **Cons**: Unconventional for directory names, potential case sensitivity issues
- **Why rejected**: Not standard practice for directory names; typically used for variables/functions

### Alternative 4: No separation (e.g., identity)
- **Description**: Using just the context name without any case convention indicators
- **Pros**: Simple, minimal
- **Cons**: Ambiguous when context names might be multi-word, inconsistent with naming practices
- **Why rejected**: Doesn't scale well for multi-word context names; lacks clarity

## Consequences

### Positive
- Consistent with common Node.js/TypeScript naming practices
- Clear and readable folder names
- Avoids case sensitivity issues across different operating systems
- Aligns with user preference from technical clarification
- Easy to implement and understand

### Negative
- Slightly more verbose than plain context names
- Requires updating any documentation or scripts that reference the folders

### Risks
- Inconsistency if some contexts use different naming
  *Mitigation*: Establish this as a strict rule and enforce through code reviews/linting
- Confusion if team members are used to different conventions
  *Mitigation*: Document the convention clearly in the repository and provide examples

## Implementation Notes
- Migration folders will be created as `src/migrations/<context_name>_context/` in snake_case
- Examples:
  - `src/migrations/identity_context/`
  - `src/migrations/billing_context/`
  - `src/migrations/product_catalog_context/`
- Update documentation and any scripts to reflect this naming convention
- Consider adding a linting rule to ensure migration folders follow the naming convention (optional)
