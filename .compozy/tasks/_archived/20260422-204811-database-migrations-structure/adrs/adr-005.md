# ADR-005: Migration Execution Approach

## Status
Accepted

## Date
2026-04-18

## Context
We need to determine how developers and DevOps engineers will execute migrations for specific bounded contexts. The options range from providing specific commands targeting each context to requiring manual navigation to the context's migration folder.

## Decision
We will provide context-specific migration commands in the package.json (e.g., `npm run migrate:identity`). This decision is based on:
- Improves developer experience by reducing the need to navigate folders
- Reduces likelihood of running migrations in the wrong context
- Aligns with the user's preference expressed during technical clarification
- Enables easy integration into CI/CD pipelines for per-context deployment
- Provides clear, discoverable commands for all team members

## Alternatives Considered

### Alternative 1: Developer navigation approach
- **Description**: Developers navigate to the context's migration folder and run migration commands manually
- **Pros**: Simple to implement, no additional configuration needed
- **Cons**: Error-prone (easy to run in wrong context), poor developer experience, not suitable for automation
- **Why rejected**: Too prone to human error; doesn't support automated pipelines well

### Alternative 2: Both approaches
- **Description**: Provide context-specific commands but also allow manual navigation
- **Pros**: Offers flexibility for different use cases
- **Cons**: Increases complexity, potential for confusion about which approach to use
- **Why rejected**: Unnecessary complexity; the command approach covers all needs

### Alternative 3: Configuration/environment variable selection
- **Description**: Use environment variables or config files to select which context to migrate
- **Pros**: Flexible, can be changed without code changes
- **Cons**: Less discoverable, risk of misconfiguration, harder to audit
- **Why rejected**: More complex than needed; explicit commands are clearer and safer

### Alternative 4: Single command with context parameter
- **Description**: One migration command that accepts a context parameter (e.g., `npm run migrate -- --context=identity`)
- **Pros**: Single command interface, flexible
- **Cons**: More complex parameter parsing, less intuitive than dedicated scripts
- **Why rejected**: Dedicated scripts are simpler and more discoverable in package.json

## Consequences

### Positive
- Clear, discoverable commands in package.json
- Reduces risk of running migrations in wrong context
- Improves developer experience and workflow
- Easy to integrate into CI/CD pipelines (e.g., `npm run migrate:identity`)
- Self-documenting; developers can see available contexts by running `npm run`

### Negative
- Requires updating package.json when adding new bounded contexts
- Slightly longer package.json scripts section

### Risks
- Scripts may become outdated if not maintained when contexts change
  *Mitigation*: Include script updates in the definition of done for adding new contexts
- Long script names if context names are lengthy
  *Mitigation*: Use consistent naming; consider aliases if needed

## Implementation Notes
- Add scripts to package.json for each bounded context:
  - `migrate:identity_context`: "typeorm migration:run -d src/migrations/identity_context/ormconfig.js"
  - `migrate:billing_context`: "typeorm migration:run -d src/migrations/billing_context/ormconfig.js"
  - etc.
- Also provide corresponding down/migration creation scripts:
  - `migrate:identity_context:down`: "typeorm migration:revert -d src/migrations/identity_context/ormconfig.js"
  - `migration:create:identity_context`: "typeorm migration:create -n MigrationName -d src/migrations/identity_context/"
- Create template ormconfig.js files in each migration folder that reference the shared database configuration
- Document the available migration commands in the README or contributing guidelines
- Consider adding a general `migrate:all` command that runs migrations for all contexts
