# ADR-001: Self-contained module structure

## Status

Accepted

## Date

2026-04-15

## Context

Each bounded context (module) must be self-contained with all necessary layers inside it, following the modular monolith principle.

## Decision

Each module under `src/modules/<name>/` contains its own:
- `domain/` — Entities, value objects, repository interfaces (ports)
- `application/` — Use cases, handlers, command/query DTOs
- `infrastructure/` — HTTP adapters, persistence, external adapters

## Folder Structure

```
src/modules/<module-name>/
├── domain/           # Pure business logic
├── application/      # Use cases orchestration
└── infrastructure/  # Adapters (http, persistence, external)
    ├── http/         # REST controllers, routes (DRIVER)
    └── persistence/ # Database adapters
```

## Consequences

- Clear ownership boundaries per bounded context
- Easy to extract module into separate service later
- Each module is independently testable
- Shared concerns extracted to `src/shared/`

## Rationale

This follows the clean-ddd-hexagonal skill reference where HTTP adapters live inside `infrastructure/http/`.
