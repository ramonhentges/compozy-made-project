# ADR-005: JWT Token Session Management

## Status

Accepted

## Date

2026-04-15

## Context

The Identity module needs session management for login/logout flows. JWT provides stateless tokens while opaque sessions require server-side state.

## Decision

Use JWT tokens for session management with the following structure:
- Access token: Short-lived (15 min), contains userId and email
- Refresh token: Longer-lived (7 days), used to obtain new access tokens

```
src/modules/identity/
  domain/
    services/
      token-service.ts    # IPasswordHasher + ITokenService ports
  infrastructure/
    http/
      adapters/
        token-adapter.ts   # JWT encoding/decoding (implements ITokenService)
```

## Alternatives Considered

### Alternative 1: Opaque session tokens

- **Description**: Server stores session in memory/database, returns random token
- **Pros**: Easy revocation, can inspect session state
- **Cons**: Requires session storage, more complex infrastructure
- **Why rejected**: Adds infrastructure complexity for MVP

### Alternative 2: In-memory mock only

- **Description**: Just scaffold the interface, defer implementation
- **Pros**: Keeps MVP focused on structure
- **Cons**: Won't have working login
- **Why rejected**: PRD requires working login/logout (User Stories)

## Consequences

### Positive

- Stateless - no server-side session storage needed
- Self-contained tokens include user info
- Easy to implement, well-understood pattern
- Standard OAuth 2.0 foundation

### Negative

- Token revocation requires token blacklist or short expiry
- Token size larger than opaque session

### Risks

- Risk: Long-lived tokens cannot be revoked immediately
- Mitigation: Use short access token expiry (15 min)
