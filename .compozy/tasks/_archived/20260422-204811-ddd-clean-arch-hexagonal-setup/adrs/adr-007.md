# ADR-007: Domain Password Hashing with Injected Adapter

## Status

Accepted

## Date

2026-04-15

## Context

Password hashing must be implemented in domain layer while following ADR-002 (zero external dependencies). The solution uses a domain port with infrastructure adapter.

## Decision

Create `IPasswordHasher` port in domain layer, implemented by `bcrypt` adapter in infrastructure:

```
src/modules/identity/
  domain/
    services/
      password-hasher.ts     # IPasswordHasher interface (port)
  application/
    use-cases/
      register-user/
        handler.ts         # Receives IPasswordHasher as dependency
  infrastructure/
    adapters/
      bcrypt-adapter.ts    # Implements IPasswordHasher using bcrypt
```

The domain service receives password as plain text, returns hashed value.

## Alternatives Considered

### Alternative 1: Plain string comparison (MVP-only)

- **Description**: Skip real hashing, just store plain password for scaffolding
- **Pros**: Fastest to implement
- **Cons**: Security risk, cannot be used in production
- **Why rejected**: Security non-negotiable even for MVP scaffold

### Alternative 2: Argon2

- **Description**: Alternative to bcrypt, more secure against GPU attacks
- **Pros**: More secure, modern standard
- **Cons**: Slower, less ecosystem support than bcrypt
- **Why rejected**: bcrypt is sufficient, widely supported

## Consequences

### Positive

- Domain remains dependency-free (follows ADR-002)
- Easy to swap hashing algorithm
- Testable with mock adapter
- Security-conscious even for MVP

### Negative

- Additional adapter layer
- bcrypt added as dependency (infrastructure only)

### Risks

- Risk: Weak hashing algorithm
- Mitigation: Use bcrypt with rounds=12 for production
