# ADR-006: pg-promise for MVP persistence scaffold

## Status

Accepted

## Date

2026-04-15

## Context

Database persistence is needed for MVP, but the domain must remain ORM-agnostic (per ADR-002). pg-promise provides lightweight SQL access while keeping domain isolated.

## Decision

Use pg-promise with the following structure:
- Domain defines `IUserRepository` port (driven port) with interface only
- Infrastructure implements repository using pg-promise
- Domain has NO knowledge of pg-promise or SQL

```
src/modules/identity/
  domain/
    repository/
      IUserRepository.ts     # Interface only (driven port)
  application/
    use-cases/
      register-user/
        handler.ts         # Uses domain entity, not DB model
  infrastructure/
    persistence/
      repositories/
        user-repository.ts # Concrete impl with pg-promise
      config/
        db-config.ts       # Database connection
```

## Alternatives Considered

### Alternative 1: Prisma

- **Description**: Type-safe ORM with schema migrations
- **Pros**: Most popular, excellent TypeScript support, migrations built-in
- **Cons**: Domain entities may leak ORM types, adds build complexity
- **Why rejected**: Keep domain agnostic per ADR-002

### Alternative 2: Drizzle ORM

- **Description**: Lightweight, SQL-like TypeScript ORM
- **Pros**: Type-safe, fast, SQL-like syntax
- **Cons**: Less established, newer ecosystem
- **Why rejected**: Deferred to Phase 2 when needed

### Alternative 3: In-memory mock

- **Description**: Just scaffold interface, implementation returns empty
- **Pros**: No dependencies, fastest MVP
- **Cons**: No actual persistence
- **Why rejected**: User Stories require "user is created" - needs actual DB

## Consequences

### Positive

- Domain stays ORM-agnostic (follows ADR-002)
- Lightweight - minimal dependency overhead
- SQL gives full control over queries
- Standard PostgreSQL support

### Negative

- Manual SQL queries - more verbose
- No built-in migrations
- Type safety requires discipline

### Risks

- Risk: SQL injection if queries built incorrectly
- Mitigation: Use parameterized queries only
