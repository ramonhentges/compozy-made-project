# ADR-004: API Calls via TanStack createServerFn Proxy

## Status

Accepted

## Date

2026-04-25

## Context

The frontend auth module must call backend endpoints: `/register`, `/login`, `/logout`, and `/token/refresh`. The PRD specifies `VITE_API_URL` for configuration. Need to choose between direct fetch, server function proxy, or other integration patterns.

## Decision

**Use TanStack createServerFn with 'use at origin' + Vite proxy**

Server functions are defined using `createServerFn({ method: "POST" })` with `api: { baseUrl: "..." }` or via Vite proxy config pointing to backend. This provides:

- **Type safety**: Server function parameters are typed end-to-end
- **SSR compatibility**: Server functions run server-side when called from loaders, client-side when called from components
- **Proxy simplicity**: Vite dev server proxies `/api/*` to backend without CORS
- **No CORS**: Same-origin proxy handles all requests

## Alternatives Considered

### Alternative 1: Direct fetch with VITE_API_URL

```typescript
const res = await fetch(`${import.meta.env.VITE_API_URL}/login`, {
  method: "POST",
  body: JSON.stringify({ email, password }),
});
```

- **Pros**: Simple, explicit, no TanStack coupling
- **Cons**: Manual type handling, no SSR optimization, requires CORS configuration on backend
- **Why rejected**: TanStack Start patterns prefer server functions; manual fetch loses type safety

### Alternative 2: Axios or Fetch wrapper

- **Pros**: Consistent interceptors, request/response transforms
- **Cons**: Additional dependency, more abstraction than needed
- **Why rejected**: TanStack Start provides sufficient functionality; add dependency only if complex requirements emerge

### Alternative 3: tRPC

- **Pros**: End-to-end type safety, excellent DX
- **Cons**: Backend must use tRPC router; backend currently uses Fastify
- **Why rejected**: Backend architectural constraint; would require significant backend changes

## Consequences

### Positive

- Type-safe: Zod schemas shared between client and server
- SSR-optimized: server functions can run in loaders without client roundtrip
- Dev experience: Vite proxy handles CORS in development
- Consistent: follows TanStack Start idioms per skill guidelines

### Negative

- Backend must be running (or Vite proxy configured)
- 'use at origin' requires careful URL configuration in production

### Risks

- **Risk**: Vite proxy not configured in production
  - **Mitigation**: Use VITE_API_URL for production; ensure CORS configured on backend
- **Risk**: Server function serialization issues with complex types
  - **Mitigation**: Keep DTOs simple, use Zod for validation

## Implementation Notes

```typescript
// frontend/src/api/auth.functions.ts
import { createServerFn } from "@tanstack/react-start/server";

export const loginFn = createServerFn({ method: "POST" })
  .validator(loginSchema)
  .handler(async ({ data }) => {
    const res = await fetch(`${getServerEnv("API_URL")}/login`, {
      method: "POST",
      credentials: "include", // Include cookies
      body: JSON.stringify(data),
    });
    if (!res.ok) throw new Error("Login failed");
    return res.json() as Promise<{ accessToken: string; user: User }>;
  });
```
