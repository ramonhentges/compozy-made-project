# ADR-003: Session Restoration via TanStack Router Root Loader

## Status

Accepted

## Date

2026-04-25

## Context

On page load or navigation to a protected route, the frontend must validate the user's session and restore their authenticated state. The PRD requires "session restoration on page refresh" with "access token refresh on 401". SSR hydration safety is mandatory to prevent hydration mismatches.

## Decision

**Use TanStack Router root loader to restore session**

The root route's `beforeLoad` or a dedicated `rootLoader`:
1. Reads the refresh token from httpOnly cookie (set by backend on login)
2. Calls a server function to exchange refresh token for new access token
3. Returns `{ accessToken, user }` to the router
4. Router data is accessible to all routes and components

```
User loads app
  → Root loader checks for refresh cookie
    → Cookie exists: call /token/refresh → store access token
    → Cookie missing: return { accessToken: null }
  → Protected route's beforeLoad checks access token
    → Valid: render page
    → Missing: redirect to /login
```

## Alternatives Considered

### Alternative 1: React Context + useEffect

```typescript
function AuthProvider({ children }) {
  useEffect(() => {
    checkSession().then(setAuth);
  }, []);
  // ...
}
```

- **Pros**: Simple, React-native
- **Cons**: Race condition between render and auth check; unprotected flash of content; not SSR-compatible
- **Why rejected**: SSR hydration mismatch, protected content visible before auth check

### Alternative 2: Middleware

- **Pros**: Runs before any route, centralized
- **Cons**: TanStack Start middleware is complex for this use case; not as idiomatic for SSR
- **Why rejected**: Over-engineered; loader pattern is simpler for this module

### Alternative 3: Client-only hydration with no SSR

- **Pros**: No hydration issues
- **Cons**: Bad for SEO, poor UX on slow connections, defeats SSR purpose
- **Why rejected**: This project uses SSR per PRD and AGENTS.md

## Consequences

### Positive

- SSR-safe: loader runs server-side, session validated before render
- Hydration-safe: auth state is part of loader data, not separate client state
- Reusable: root loader data available to all child routes via `useRootLoaderData()`
- Redirect-friendly: loader can return redirect response for protected routes
- On-demand refresh: if access token expires, component calls refresh server function

### Negative

- All protected routes depend on root loader; root loader failure blocks entire app
- Requires backend /token/refresh endpoint

### Risks

- **Risk**: Root loader always runs, even for public pages (performance)
  - **Mitigation**: Token refresh is a lightweight operation; can be skipped if no cookie
- **Risk**: Hydration mismatch if store and loader disagree
  - **Mitigation**: Zustand store initialized empty, populated from loader data

## Implementation Notes

```typescript
// frontend/src/routes/__root.tsx
export const useRootLoaderData = createRootRoute();

export const routeConfig = useRootLoaderData.to({
  loader: async () => {
    const cookie = getCookie('refreshToken');
    if (!cookie) return { accessToken: null, user: null };
    // Call refresh server function
    const { accessToken, user } = await refreshSession();
    return { accessToken, user };
  },
});
```
