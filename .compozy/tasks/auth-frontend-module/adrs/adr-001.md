# ADR-001: Auth Frontend Module V1 Scope

## Status

Accepted

## Date

2026-04-25

## Context

Building a frontend authentication module with registration, login, and protected routes using TanStack Start + shadcn + Tailwind. Backend has existing auth API (email/password registration, JWT login). User requests full auth flow in V1 including registration.

Council debate focused on: (1) registration timing, (2) CSRF protection complexity, (3) SSR cookie edge cases.

## Decision

**V1 scope: Registration + Login + Protected Routes with basic security controls**

1. Registration page (email, password, name) with Zod validation
2. Login page with JWT token response
3. httpOnly cookie for refresh token, memory storage for access token
4. Route guards protecting home page
5. Basic security: SameSite cookies, generic error messages
6. Logout clearing cookies

**Defer to V2:**
- CSRF token implementation (use SameSite=Strict for now)
- Rate limiting (backend concern)
- Email verification
- Password reset

## Alternatives Considered

### Alternative 1: Login-only V1

- **Description**: Defer registration to V2, ship only login
- **Pros**: Reduced complexity, faster time-to-market
- **Cons**: User explicitly requested registration; blocks user acquisition
- **Why rejected**: User requirement for full auth flow

### Alternative 2: Full security in V1

- **Description**: Include CSRF tokens, rate limiting, email verification in V1
- **Pros**: Maximum security from day one
- **Cons**: Significant complexity increase; CSRF tokens add full-page form handling complexity
- **Why rejected**: Scope creep; SameSite=Strict provides reasonable CSRF protection for V1

## Consequences

### Positive

- Users can register and login in a single module
- httpOnly cookie pattern provides XSS protection for refresh token
- Route guards provide UX-appropriate redirects

### Negative

- SameSite=Lax required for cross-origin login API calls (trade-off between CSRF protection and usability)
- SSR hydration edge cases require careful implementation

### Risks

- **Risk**: SSR hydration mismatch on auth state
  - **Mitigation**: Fetch session server-side in root loader, pass via context
- **Risk**: CSRF vulnerability with SameSite=Lax
  - **Mitigation**: Plan CSRF token implementation for V2; document limitation

## Implementation Notes

- Backend registration API needs `name` field added to request/response
- TanStack Start loader-based auth validation pattern for protected routes
- Use `@tanstack/react-form` + Zod + shadcn Form components
