# ADR-002: Access Token Storage

## Status

Accepted

## Date

2026-04-25

## Context

The auth frontend module requires storing the JWT access token in a way that prevents XSS theft while remaining SSR-compatible. The PRD specifies "memory storage" but does not mandate the implementation mechanism.

## Decision

**Use Zustand store with client-only persistence middleware**

The access token is stored in a Zustand auth store using `useAuthStore()`. The store:
- Holds `{ accessToken: string | null, user: User | null }`
- Provides `setAccessToken(token)` and `clearAuth()` actions
- Uses `skipHydration` to prevent SSR state leakage
- Initialized empty, populated via root loader or login response

## Alternatives Considered

### Alternative 1: Module-level variable (singleton)

```typescript
// auth-token.ts
let accessToken: string | null = null;
export const getToken = () => accessToken;
export const setToken = (t: string) => { accessToken = t; };
```

- **Pros**: Zero abstraction, fastest access
- **Cons**: Hard to test, SSR hydration issues, no React integration, module-level state persists across HMR
- **Why rejected**: Poor testability, SSR complexity

### Alternative 2: React Context

```typescript
const AuthContext = createContext<{ token: string | null }>({ token: null });
```

- **Pros**: React-native, component-scoped
- **Cons**: Prop-drilling without hooks, no cross-cutting store pattern
- **Why rejected**: Zustand is project standard per AGENTS.md; Context is better for theme, not auth state

### Alternative 3: sessionStorage

- **Pros**: Survives page refresh
- **Cons**: Vulnerable to XSS (same origin), not true memory-only
- **Why rejected**: Memory storage requirement in PRD; XSS risk

## Consequences

### Positive

- SSR-safe: store starts empty, populated after hydration
- Project standard: Zustand is mandated in AGENTS.md
- Testable: can mock store in tests
- DevTools: Zustand devtools middleware shows state changes
- No prop drilling: hooks pattern `useAuthStore()` is idiomatic

### Negative

- Store persists in memory only; refreshing page clears token (intentional per "on-demand refresh" requirement)

### Risks

- **Risk**: Store state leaks to SSR
  - **Mitigation**: Use `skipHydration` option, initialize empty
- **Risk**: Token lost on page refresh
  - **Mitigation**: Root loader reads refresh cookie and restores session

## Implementation Notes

```typescript
// frontend/src/stores/auth.store.ts
interface AuthState {
  accessToken: string | null;
  user: { id: string; email: string; name: string } | null;
}

export const useAuthStore = create<AuthState>((set) => ({
  accessToken: null,
  user: null,
}));
```
